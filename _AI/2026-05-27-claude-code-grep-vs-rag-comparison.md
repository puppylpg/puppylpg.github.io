# 两种技术路线的碰撞

同样是 AI 编程工具的代码检索问题，两篇文章给出了截然不同的答案：

| 代表 | Claude Code（原生） | Claude Context（第三方增强） |
|------|---------------------|------------------------------|
| **技术路线** | grep + glob + read | RAG（向量检索 + 语义理解） |
| **核心理念** | 让模型自己决策 | 用工程手段帮模型准备材料 |
| **设计哲学** | 信任 LLM 的能力，把决策权还给它 | LLM 还不够强，需要预建索引辅助 |

# Claude Code 的 grep 派：为什么不用 RAG？

## RAG 在代码场景的五大痛点

| 痛点 | 说明 |
|------|------|
| **代码切不动** | 代码有严格结构，拦腰切断会导致上下文丢失（函数分成两半，if/else 分家） |
| **向量干不了精确匹配** | 找 `getUserById` 会返回一堆"相似"函数：`getUserByName`、`getUserByEmail`... |
| **索引跟不上变化** | 代码每天都在改，重建索引成本高，不重建就用旧数据 |
| **冷启动慢** | 百万行代码建索引要十几分钟，用户等不起 |
| **黑盒不可解释** | 召回错了不知道为什么，调试困难 |

## Claude Code 的三件套

| 工具 | 作用 | 核心特性 |
|------|------|----------|
| **Glob** | 按文件名找文件 | 结果按修改时间倒序，最近改过的排前面 |
| **Grep** | 按内容关键字找代码 | 三种输出粒度：content / files_with_matches / count |
| **Read** | 按需读文件内容 | 默认只读 2000 行，每次实时读磁盘，不缓存 |

## 核心优势：多轮迭代循环

**RAG 的范式**：「考试发卷子」→ 一次性召回 Top-K → 一次性生成答案。没有循环，没有反悔。

**Claude Code 的范式**：「现场探案」→ Grep → 看结果 → Read → 看内容 → 「找到了」。

**关键能力**：每个回合都能根据上一步结果调整方向：
- Grep 结果是空的？改关键字再搜
- Read 出来的代码不像你想的？再 Grep 相关函数
- 文件引用了另一个文件？跟过去看下那个文件

当任务复杂时，还会**派子 agent 去探索**，隔离上下文污染。

# Claude Context 的 RAG 派：grep 有什么问题？

## grep 的三大问题

| 问题 | 说明 |
|------|------|
| **信息过载** | grep 搜索结果中 99% 为无关噪音，AI 易被洪流淹没 |
| **语义盲目** | 仅匹配字符组合，找不到 `compute_final_cost()` 与 `calculate_total_price()` 的语义关联 |
| **上下文失忆** | 仅返回匹配行，缺失函数所属类、依赖关系等关键上下文 |

**实际案例**：让 Claude Code 找 bug，反复 grep + read 5 分钟后才定位，但读取的代码 99% 无关。

## Claude Context 的解决方案

| 痛点 | 解决方案 |
|------|----------|
| 信息过载 | 向量相似度排序，最相关的代码片段置顶 |
| 语义盲目 | embedding 理解代码语义，函数名不同但功能相似也能匹配 |
| 上下文失忆 | 每个搜索结果包含完整的代码上下文（类、函数、依赖） |

## 核心技术：如何解决 RAG 的痛点？

### 代码切分

| 策略 | 说明 |
|------|------|
| **AST 抽象语法树**（主策略） | 按语义单元切分，每个 chunk 都是完整的语法单元 |
| **LangChain 文本切分**（兜底） | 对于 AST 无法处理的语言，按字符数量切分 |

### 代码变动同步

使用 **Merkle Tree**（哈希指纹树）：

1. **快速检测**：计算根哈希，对比上次快照（几毫秒）
2. **精确对比**：文件级别对比，找出变化的文件
3. **增量更新**：只对变化的文件重新计算向量

# 深层对比：设计哲学的差异

## Claude Code 的潜台词

> **LLM 已经足够强，工程的角色是给它准备好工具，把决策权还给它。**

grep 不替模型做任何决定，它只是个工具。用还是不用、什么时候用、怎么用，全是模型说了算。

**押注的是「模型会越来越强」——长期主义。**

## Claude Context 的潜台词

> **LLM 还不够强，用工程手段帮它把材料准备好。**

chunking、embedding、向量召回，本质都是「替模型做决定」。

**当前效率优先，用工程补齐模型的短板。**

# 各自的适用场景

## grep 方案（Claude Code 原生）适合

- ✅ 单项目、探索式开发
- ✅ 需要精确性（找特定函数名）
- ✅ 要求实时性（代码刚改就能搜到）
- ✅ 零冷启动、零配置

## RAG 方案（Claude Context）适合

- ✅ 巨型代码库（千万行级别）
- ✅ 纯语义查询（「找认证相关代码」）
- ✅ 需要降低 Token 成本
- ✅ 多人协作知识库（代码 + 文档 + Wiki 混合检索）

# 我的思考：可以共存吗？

**答案是可以。**

两篇文章其实指向同一个结论：**没有银弹，场景决定选择。**

如果把它们结合起来：
- **简单定向搜索**（找特定函数）→ 用 grep，精确快速
- **开放式探索**（理解整个认证模块）→ 用 RAG，语义召回
- **多轮迭代** → 让模型自己判断用哪种方式

Claude Code 已经有了「派子 agent 去探索」的机制，未来完全可以在子 agent 里集成 RAG 能力，实现混合检索。
