# 总结文章到博客 post

## 总体架构

主 agent 不自己抓原文，派子 agent 做「抓正文 + 写总结 + 落盘」，主 agent 只接收三行结果，再按 `AGENTS.md` 执行预览和 git 操作。

```
主 agent
  ├── 确定发布目标（默认 _ai/，用户可指定 tech/post/tutorial）
  ├── 执行 date 获取时间
  ├── spawn 子 agent: 抓原文 + 写总结 + Write 到 _<target>/
  │       │
  │ <──── PATH + TITLE + COMMIT_DESC
  │
  └── 后续按 AGENTS.md 中的「文章发布通用流程」执行预览、确认、git 发布/清理
```

> 文章发布通用流程（目标目录、本地预览、用户确认、git 操作、清理重写）见项目根目录 `AGENTS.md`。

## 工作流程

### 1. 确定发布目标

按 `AGENTS.md` 中的「目标目录与默认分类」执行：

- 默认 → `_ai/`，默认分类 `[ai]`
- tech / post → `_posts/`，默认分类 `[tech]`
- tutorial → `_tutorials/`，默认分类 `[tutorial]`

### 2. 派子 agent 抓取 + 总结 + 写文件

派 agent 前先执行 `date '+%Y-%m-%d %H:%M:%S %z'` 拿到实际日期和时间。**不得硬写时间**。

使用当前环境支持的子 agent / 子任务机制执行，prompt 使用 `assets/PROMPT.md` 中的模板。

无论首次生成还是按反馈重写，都必须遵循根 `AGENTS.md` 的**零上下文读者原则**：来源文章、用户追问和修改意见只提供素材，成文不能假设读者见过这些上下文。主 agent 验收时要检查章节是否按概念依赖顺序展开，而不只检查单段措辞。

派子 agent 时必须传入：

- 原文来源 URL
- 当前日期时间
- **目标目录**（`_ai/`、`_posts/` 或 `_tutorials/`）
- **默认分类**（`[ai]`、`[tech]` 或 `[tutorial]`）

### 3. 预览与发布

子 agent 返回 `PATH`、`TITLE`、`COMMIT_DESC` 后，主 agent 按 `AGENTS.md` 中的「文章发布通用流程」执行：本地预览 → 用户确认 → 询问后 git commit/push → 或清理/重写。

重写时：派新子 agent，prompt 带上原始来源、现有文件路径、修改意见、目标目录、默认分类、同样的写作和 frontmatter 规则。

### 4. 后续问答转 FAQ

如果总结文章生成后，用户围绕这篇文章继续追问，处理方式取决于用户要求：

- 用户明确要求“FAQ”“问答整理”或“把问题加到 FAQ”时，直接编辑现有文章并使用下面的 FAQ 规则。
- 用户只要求“补充正文”“把解释加到文章”时，不要自动转成 FAQ。把新增信息拆解后放回主线中对应概念的逻辑位置，并按需要重写相邻标题和过渡，使零上下文读者可以连续阅读。

FAQ 模式遵循：

- 将 FAQ 追加到全文最后；若文章已有 `# 核心`、`# 评价`，FAQ 放在它们之后，并用 `---` 分隔。
- 使用 `# FAQ：<主题>` 作为顶层标题，问题用 `## 1. ...`、`## 2. ...` 编号。
- 按概念类别和读者理解顺序重组问题，不按聊天时间线逐条搬运。
- 答案吸收对话中已澄清的最终表达，保留关键例子、边界条件、对比和必要的 Mermaid 图。
- 只追加 FAQ，不改动原文章主线，除非用户明确要求重写正文。
- 完成后按 `AGENTS.md` 预览并等待用户确认。

## 常见错误

- ❌ 主 agent 自己抓取原文
- ❌ 微信公众号用常规抓取工具（必须用浏览器自动化）
- ❌ 子 agent 把全文 markdown 塞回返回值
- ❌ 文件名用中文
- ❌ title 照搬原文标题
- ❌ 英文文章的晦涩专业术语首次出现时不附英文原词
- ❌ 默认发布到错误目录（未说明时应发到 `_ai/`）
- ❌ 后续 FAQ 插到 `# 核心` / `# 评价` 前面，破坏文章原有收束（默认放全文最后）
- ❌ 把用户追问的答案直接追加到正文，形成对不可见问题的回答
- ❌ 只润色新增段落，没有检查它依赖的对象、概念和关系是否已经在前文建立
