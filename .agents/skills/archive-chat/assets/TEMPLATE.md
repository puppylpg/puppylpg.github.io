# 对话总结文章模板

## Frontmatter

```yaml
---
title: "核心对象 + 对比维度或核心问题"
date: YYYY-MM-DD HH:MM:SS +0800
categories: [ai]
tags: [tag1, tag2, tag3]
description: "一句话摘要"
---
```

- `categories` 根据发布目标选择：
  - 默认 `_ai/` → `[ai]`
  - `_posts/`（tech）→ `[tech]`
  - `_tutorials/` → `[tutorial]`
  - 有更细分领域可追加，例如 `[ai, tools]`、`[tech, tools]`、`[tutorial, docker]`
- `tags` 从关键词中提取 3-8 个
- 不写 `layout:` 或 `last_modified_at`
- `categories` 和 `tags` 必须小写

## 标题要求

- 默认采用直接、技术性、可检索的陈述式标题，优先使用“核心对象 + 对比维度/核心问题”结构。
- 标题必须包含核心主题关键词，脱离正文也能判断文章内容。
- 除非用户明确要求创意风格，否则不要使用反问、悬念、隐喻或“一文看懂”“揭秘”“到底”等营销号句式。
- 不要把局部例子、类比、图表细节或俏皮话当成全文标题。
- 内部生成至少 3 个候选，按主题清晰度、可检索性、范围准确性和简洁自然排序，默认选择最朴素、最准确的候选。

## 正文组织原则

**不强制固定骨架**，文章结构应按内容本身的逻辑层次灵活编排，核心原则是：

- **按知识内在逻辑递进**，而非对话时间线
- **面向零上下文读者**：对话只提供素材，正文不能假设读者问过问题、看过回答或知道此前纠正过什么
- **按依赖顺序展开**：对象与背景 → 组件与关系 → 操作或现象 → 内部原理 → 边界与结论；追问答案拆开后放回对应概念第一次需要它的位置
- **不要结论先行**：开头先写背景、问题张力、读者困惑和推演路径，最终判断要从正文推出来
- **先铺垫基础概念，再展开实操，最后讲原理**
- 让读者能独立阅读并理解，不需要回看对话上下文

参考范例（良好逻辑链）：`问题背景 → 直觉例子 → 前置概念 → 推演过程 → 阶段性小结 → 工程含义 / 实用技巧`

## 正文骨架（仅供参考，不强制）

TOC 必须严格是下面这两行：

```markdown
1. Table of Contents, ordered
{:toc}
```

之后的标题层级按内容逻辑自由安排。常见模式包括：

- **问题驱动**：背景 → 核心概念 → 解决方案 → 踩坑与技巧
- **原理驱动**：基础原理 → 配置实践 → 进阶场景 → 总结
- **对比驱动**：方案 A vs 方案 B → 选型建议 → 实操步骤
- **推演驱动**：问题背景 → 直觉例子 → 概念拆解 → 为什么成立 → 推出的结论

## 正文要求

- 不要大段粘贴原始对话，用第三人称叙述体重新组织
- **按知识内在逻辑重新编排内容，不要按对话时间线组织**
- 不要使用依赖聊天上下文的答疑式过渡，例如没有明确前文铺垫的“真正的问题是”“容易误解”“先停一下”“答案是”“还剩最后一步”
- 完稿后只读标题和每节首句做冷读检查：读者应能判断每节讲什么、为什么出现在这里、依赖哪些已介绍概念
- 开头不要直接写“结论是”“答案是”“核心观点是”。先让读者知道问题从哪里来、为什么容易误解、需要补哪些概念，再进入判断。
- 代码块、命令行等关键输出保留核心片段
- 突出"为什么这样做"和"结果是什么"
- 保留清晰原理阐述的结构化形式：层次化原理解释（执行链条、调用栈、架构层次对比等）保留层级列表、流程图或对比表格，不要为压缩字数改扁平
- 外部链接必须使用 Markdown 内联链接，嵌入到有意义的文字中；不要在正文里裸贴 URL。引用资料时说明它支撑了哪个判断，优先引用官方文档、项目主页或一手来源。
- 总长度控制在 500-2000 字，除非对话本身很长
