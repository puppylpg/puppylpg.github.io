# 对话总结与发布

## 总体架构

```
主 agent
  ├── 回顾对话，提炼核心信息
  ├── 设计文章逻辑骨架，补齐必要前置概念
  ├── 自动生成 title / tags / description
  ├── 确定发布目标（默认 _ai/，用户可指定 tech/post/tutorial）
  ├── 写文件到对应目录
  └── 后续按 AGENTS.md 中的「文章发布通用流程」执行预览、确认、git 发布/清理
```

> 文章发布通用流程（本地预览、用户确认、git 操作、清理重写）见项目根目录 `AGENTS.md`。

## 工作流程

### 1. 回顾对话，提取核心信息

从当前会话上下文中提炼：

- **主题**：这次对话围绕什么展开（一句话概括）
- **主要步骤**：按逻辑层次（而非对话时间线）列出关键阶段
- **关键指令与输出**：用户的核心请求、AI 执行的关键操作、关键输出结果
- **最终结果**：解决了什么问题、学到了什么、最终成果。这里只是素材提取，不能理解为正文开头要先写结论。
- **清晰原理阐述**（如有）：对话中用户认可的层次化原理解释，保留结构化形式
- **前置概念链路**：读者理解主题前必须先知道什么概念，以及这些概念之间如何递进
- **关键推理链**：对话中反复澄清的“为什么”，例如为什么这样定义、为什么这个公式能表达该概念、为什么某个优化成立或失效
- **关键类比/例子**：用户认可或反复强调的形象例子、反例、纠错点，不要因为压缩篇幅省略
- **标题范围**：最终文章实际覆盖的范围是否已经超过初稿标题，必要时重新命名
- **技术栈/关键词**：涉及的技术、工具、概念，用于生成 tags

### 2. 设计文章逻辑骨架

文章不是对话纪要，也不是简单压缩。骨架设计必须分三层自顶向下完成，**不得跳层直接写正文**：

#### 第一层：宏观分块

先从最高视角判断：对话涵盖几个独立主题？每个主题能否独立成节？常见模式：

- **单主题深入**：一个概念从基础到应用，线性展开
- **两类场景对比**：同一技术在不同场景下的不同用法（如“验收 vs 爬虫”）
- **问题驱动递进**：一个问题引出下一个问题，链式展开

在这一步确定顶层章节（H2），明确每章的核心一句话：它在讲什么、为什么在这里。

#### 第二层：逻辑内容整理

对每个宏观块，整理其内部逻辑顺序：

- 默认不要结论先行。文章开头应先交代背景、问题张力、读者为什么会困惑，以及接下来要沿着什么推演路径展开；除非用户明确要求 TL;DR，否则不要在开头直接盖最终判断。
- 结论必须从前面的背景、概念、例子和推理中自然推出。可以在文末或对应小节末尾总结，但不要用“总之/结论是/答案是”替代推演。
- 从读者最容易理解的直觉或具体例子开始，再进入抽象概念。
- 技术主题优先使用“直觉例子 → 公式 → 图形化解释 → 工程含义 → 常见误区”的链路。图形不应默认全用流程图，要按概念本身选择流程图、时序图、结构图、状态图、架构块图、函数曲线等不同形态。
- 如果目标概念依赖前置概念，先补齐前置概念。
- 同一概念如果经历了用户纠偏，最终文章应吸收纠偏后的表达，而不是保留初稿里的生硬顺序。

#### 第三层：写正文

骨架确定后再下笔。此时每个小节的目的和顺序已经清楚，写作只是填充。

- 保留关键“为什么”推理链，不要因为压缩篇幅省略。
- 原文信息丰富时，以逻辑清晰和可读性为优先，宁可扩展为长文，也不要删掉关键推导、类比、例子。

### 2.5. FAQ 模式

当用户说“把聊天转成 FAQ”“把刚才的问题加到 FAQ”“把上面的问答整理进文档”等类似请求时，使用 FAQ 模式。FAQ 不是对话逐句搬运，而是把讨论中反复澄清的问题按知识逻辑归类，改写成读者会自然提出的问题。

有两种处理方式：

- **追加到现有文章**：如果当前对话围绕一篇已生成或已知路径的文章展开，优先编辑该文章，把 FAQ 追加到全文最后。若文章已有 `# 核心`、`# 评价`，FAQ 放在它们之后，并用 `---` 分隔；不要插到核心和评价之前，除非用户明确要求。
- **生成独立 FAQ 文章**：如果没有明确目标文章，按普通归档流程新建文章，但正文以 FAQ 为主。

FAQ 写法：

- 顶层标题使用 `# FAQ：<主题>`，问题用 `## 1. ...`、`## 2. ...` 编号。
- 按概念类别排序，不按聊天时间线排序。
- 问题标题要短、自然、具体，避免“用户问了什么”这种纪要口吻。
- 答案要吸收对话中已经澄清的最终表达，必要时保留关键例子、对比表、Mermaid 图。
- 对误解和边界条件要单独成问，例如“是不是一个工具对应一个工作区？”“分错工作区怎么办？”。
- FAQ 追加到现有文章时，不改动原文章主线，除非用户要求同步重写正文。
- 修改后仍按 `AGENTS.md` 执行预览和确认流程。

### 3. 确定发布目标

按 `AGENTS.md` 中的「目标目录与默认分类」执行：

- 默认 → `_ai/`，`categories: [ai]`
- tech / post → `_posts/`，`categories: [tech]`
- tutorial → `_tutorials/`，`categories: [tutorial]`

### 4. 生成元数据

根据对话内容直接生成 title、tags、description，不需要向用户确认。

执行 `date '+%Y-%m-%d %H:%M:%S %z'` 获取当前时间。**不得硬写时间**。

标题应匹配最终文章范围。若文章从单点问题扩展为完整知识链，应同步调整标题，避免标题只覆盖初稿主题。

#### 标题生成规则

默认生成**直接、技术性、可检索的陈述式标题**，不要把“吸引眼球”当作优化目标。

- 标题前半段优先写清核心对象，后半段写对比维度、核心问题或文章动作。常用结构是“核心对象 + 对比维度/核心问题”，例如“OpenAI、Anthropic 与国产旗舰模型 API 价格对比”。
- 标题必须包含读者会搜索的核心名词。脱离正文只看标题，读者也应能判断文章讨论什么。
- 除非用户明确要求故事化、公众号风格或创意标题，否则禁止默认使用反问、悬念、隐喻和营销号句式，例如“到底”“一文看懂”“揭秘”“为什么说”以及与核心主题无关的修辞钩子。
- 不要把正文中的局部例子、类比、图表细节或一句俏皮话提升为整篇标题，除非它确实概括全文。
- 在内部生成至少 3 个候选标题，依次按**主题清晰度、搜索关键词、范围准确性、简洁自然**评估。默认选择最朴素、最准确的候选，而不是最有悬念的候选；候选过程无需展示给用户。
- 写完正文后再次自检标题：是否准确覆盖全文，是否把核心关键词放在前面，是否存在删掉后信息不受损的修辞。如果存在，改成更直接的标题。

### 5. 生成 Markdown 文件

文件保存到对应目录：`_<target>/YYYY-MM-DD-<slug>.md`。

- slug 规则见 `AGENTS.md`。
- 正文格式和结构要求见 `assets/TEMPLATE.md`。

### 6. 预览与发布

按 `AGENTS.md` 中的「文章发布通用流程」执行：本地预览 → 用户确认 → 询问后 git commit/push → 或清理/重写。

## 常见错误

- ❌ 写文件前停下来确认 title / tags / description
- ❌ 为了吸引眼球使用反问、隐喻、悬念或营销号式标题，导致核心主题和搜索关键词后置
- ❌ 把局部例子、类比或图表细节提升为全文标题
- ❌ 文件名用中文
- ❌ 大段粘贴原始对话而不重新组织
- ❌ 硬写日期时间而不执行 `date` 命令
- ❌ 把结构化原理阐述扁平化成叙述体
- ❌ 按对话时间线组织文章（应改为按逻辑层次重新编排）
- ❌ FAQ 按聊天顺序逐条罗列，而不是按读者问题和概念类别重组
- ❌ 把 FAQ 插到 `# 核心` / `# 评价` 之前，打断原文章收束（应默认追加到全文最后）
- ❌ 默认发布到错误目录（未说明时应发到 `_ai/`）
- ❌ 为了缩短篇幅，丢掉用户反复强调的关键推理、类比、例子和纠错点
- ❌ 结论先行：开篇直接给判断、答案或总结，却没有先给背景、问题张力、前置概念和推演路径
- ❌ 一上来抛高级概念或公式，没有先解释读者必须理解的前置概念
- ❌ 只写公式，不解释符号、维度，以及为什么这个公式表达了该概念
- ❌ 讲函数或公式时只给公式和表格，不画函数图像、分布图或参数变化图辅助理解
- ❌ 把图表集中堆到单独章节，而不是放在对应概念第一次出现、最需要解释的位置
- ❌ 对矩阵运算只写公式，不用图表说明某个格子的含义
- ❌ 大段文字堆叠，缺少小标题、表格、图、结论块等视觉层次
- ❌ 文章范围已经变化，但标题仍沿用初稿标题

## 正文与可视化要求

正文格式和结构要求见 `assets/TEMPLATE.md`。补充一点：

引用外部资料时，必须使用 Markdown 内联链接，把链接嵌入到有意义的文字中，例如 `[OpenAI Codex 的 AGENTS.md 文档](https://developers.openai.com/codex/guides/agents-md)`。不要把裸 URL 单独贴在正文里。需要增强可信度时，应优先引用官方文档或项目主页，并在正文中解释来源支持了哪个判断。

当对话内容涉及流程、层级、对比、时序等结构时，**优先使用 Mermaid 图**（`mermaid` 代码块）替代 ASCII 示意图或纯文本层级图。博客主题已原生支持 Mermaid 渲染。

Mermaid 选型要服务理解，不要默认所有图都使用 `flowchart`。同一概念如果有多个视角，可以保留多张互补图：例如先用 `flowchart` 表达依赖、映射、数据流，再用 `block-beta` 表达架构分层、模块布局、内存/缓存块形状。多图必须各自承担不同解释任务，避免为了“有图”而重复装饰。

当对话内容涉及函数、公式或参数变化时，优先画图辅助理解，而不是只给公式和数值表：

- 单变量函数（如 Sigmoid、指数函数、损失函数）优先画函数曲线，标出关键点、上下界、增长/饱和趋势。
- 向量到分布的函数（如 Softmax）优先画概率分布图、柱状图或热力图，而不是强行画成单变量曲线。
- 参数会改变函数形态或输出分布时（如 temperature、阈值、正则化系数），优先画多组参数对比图。
- 图应放在对应概念第一次深入解释的位置。例如指数函数图放在“指数放大”小节，Sigmoid 曲线放在“Sigmoid 在做什么”小节，temperature 分布图放在“温度参数”小节。
- 图表顺序必须服务文章逻辑，不要因为“有图”而打断原来的概念递进。

常用场景与图表类型：

| 场景 | 推荐图表 |
|------|---------|
| 多步骤流程 / 决策分支 | `flowchart TD`（纵向）或 `flowchart LR`（横向） |
| 层级 / 架构分层 | `flowchart` + `subgraph` 划分子系统 |
| 系统架构 / 模块布局 / 内存块形状 | `block-beta`，或 `flowchart` + `subgraph` |
| 时序交互 / 调用链 | `sequenceDiagram` |
| 状态机 / 生命周期 | `stateDiagram-v2` |
| 接口继承 / 类关系 | `classDiagram` |
| 数据实体 / 表关系 | `erDiagram` |
| 时间演化 / 里程碑 | `timeline` 或 `gantt` |
| 知识树 / 概念分支 | `mindmap` |
| 简单占比 | `pie` |
| 二维取舍 / 象限分析 | `quadrantChart` |
| 矩阵乘法 / 注意力得分 | Markdown/HTML 表格，高亮关键格子 |
| 单变量函数 / 激活函数 | SVG/HTML 函数曲线，标出关键点和趋势 |
| 向量归一化 / 概率分布 | SVG/HTML 柱状图、热力图或表格 |
| 参数影响 / 温度变化 | 多组曲线或多组柱状图对比 |
| 概念角色对比 | Markdown 表格 |
| 阶段性小结 | Markdown 引用块或醒目的短小节，放在推演之后 |

样式约定：
- 用 `style` 给关键节点上色，区分不同路径或状态（如 `#e3f2fd` 正常流程、`#e8f5e9` 成功/直连、`#fff3bf` 警告/中间环节、`#ffe3e3` 错误/阻塞）
- 避免纯装饰性节点，每个节点和箭头都应承载信息
- 单张图只表达一个核心关注点，复杂流程拆成多张

数学与代码约定：
- 需要数学公式时使用 LaTeX 公式块或行内公式，并先用直觉解释铺垫，再给出公式。
- 公式出现后要定义关键符号，必要时说明矩阵形状和每个维度代表什么。
- 涉及矩阵乘法时，优先展示“某个格子”的计算含义，例如某个 token 对另一个 token 的注意力分数。
- 代码用于教学时，应添加足够细的注释；关键演示代码可以逐行解释，但注释应放在被解释代码的上一行，不要写成行尾 inline comment。
